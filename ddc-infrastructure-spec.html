§ 01

The DDC Model Outside is chaos. Inside is determinism. The boundary is the event.

A VM inside any computer is a controlled execution layer: its own state, its own rules, its own input stream — isolated from the host. The DDC model maps this to a physics metaphor drawn from lava lamps: excitement accumulates into density, density drives convergence, and time causes decay.

inside / determinism

🌋

The lava lamp principle: Entropy happens outside the VM. It is captured as an event at the boundary. Once inside, it becomes fixed, immutable, and replayable. outside(chaos) → event → inside(determinism)

The Five-Layer Reduction

🌍

Host System

Physical machine. Non-deterministic. Source of entropy. Network, time, user input — all exist here before capture.

📦

VM (Module)

Deterministic runtime. Isolated state. Its only source of truth is its own eventLog. No hidden channels.

📋

eventLog

Append-only, immutable. The VM's memory. Every state is a pure projection of this log from position 0 to n.

⚙️

deriveState

Pure function. No I/O. state = f(eventLog[0…n]). The VM's CPU. Same log always produces same state.

📡

render / effects

Outputs only. DOM, network, metrics. Never feeds back into state directly — they emit new events at the boundary.

Core Invariant
VM_stateₙ = deriveState(eventLog[0…n])
Nothing else is allowed to influence state. No globals. No ambient reads. No shared memory.

§ 02

Signal Physics Energy · Decay · Priority — the three observables of a live node.

Every module in the DDC system exposes three computed scalar signals, derived purely from its eventLog. Together they describe whether a node is hot, cooling, or ready to converge with another.

Energy

Energy measures how active a module is right now relative to its total history. It rises when domain-primary events fire, and normalises against log length to stay bounded.

Energy Formula
energy = count(primaryEvents, window=100) / max(1, totalEvents)
Range [0, 1]. Clipped. Each module defines what counts as a "primary" event in its schema.

Example — Task Manager: primary events are TaskCreated and TaskCompleted. A burst of completions spikes energy. A quiet period after a burst lets it fall.

Decay

Decay measures recency — how long since the last significant activity. A freshly-active node has decay near 1.0. A stale node trends toward 0.

Decay Formula
decay = 1 − (eventsSince_lastPrimary / log_length)
Range [0, 1]. As time passes without activity, eventsSince grows and decay falls.
❄️

This is the lava lamp's cooling mechanism. When a blob receives no excitement, its density falls (decay → 0) and it separates from active blobs. When re-excited, density rises and it can converge again.

Priority

Priority is the composite scheduling weight — used by routers, queues, and orchestrators to decide which module to wake, which VM to schedule, which node to route to. Each module may tune its own weighting coefficients in the schema.

Priority Formula (default)
priority = (energy × 0.6) + ((1 − decay) × 0.4)
Coefficients are tunable per module: some are decay-dominant (0.5/0.5), some energy-dominant (0.7/0.3).
Module type Energy weight Decay weight Rationale
Task Manager 0.6 0.4 Active work matters more than age
Messaging 0.5 0.5 Recency equally important
CRM 0.6 0.4 Activity-weighted
E-Commerce 0.7 0.3 Order bursts are the signal
Identity 0.5 0.5 Staleness is a security signal

Convergence

Two VMs converge when their combined priority exceeds a routing threshold and their correlationId namespaces overlap. This is not a merge — each VM keeps its own log. Convergence means they start exchanging events via the router.

Convergence Condition
converge(VM_a, VM_b) iff priority_a > θ AND priority_b > θ AND overlap(correlationSpace_a, correlationSpace_b)
θ is the convergence threshold, typically 0.5. Overlap means they share a common causal ancestor.

§ 05

Signal Functions — Implementation How energy, decay, and priority are computed from the eventLog at runtime.

Signal functions are pure functions that take an eventLog array and return a scalar in [0, 1]. They are called by the orchestrator on every scheduler tick and by the router to make convergence decisions.

signal_functions.js — canonical runtime implementation JS 
/**
 * Compute energy for a module's eventLog.
 * energy = how much primary-event activity has occurred recently.
 */
function computeEnergy(eventLog, primaryTypes, window = 100) {
  const total    = eventLog.length;
  if (total === 0) return 0;
  const slice     = eventLog.slice(-window);
  const primary   = slice.filter(e => primaryTypes.has(e.type)).length;
  const denom     = Math.max(1, total);
  return Math.min(1.0, Math.max(0.0, primary / denom));
}

/**
 * Compute decay for a module's eventLog.
 * decay = recency — falls toward 0 as the log goes quiet.
 */
function computeDecay(eventLog, primaryTypes) {
  const total  = eventLog.length;
  if (total === 0) return 0;
  let lastPos = -1;
  for (let i = total - 1; i >= 0; i--) {
    if (primaryTypes.has(eventLog[i].type)) { lastPos = i; break; }
  }
  if (lastPos === -1) return 0;
  const eventsSince = total - 1 - lastPos;
  const decay       = 1.0 - (eventsSince / Math.max(1, total));
  return Math.min(1.0, Math.max(0.0, decay));
}

/**
 * Compute priority — the composite scheduling weight.
 * Weights are module-defined in the schema.
 */
function computePriority(energy, decay, w = { e: 0.6, d: 0.4 }) {
  return Math.min(1.0, (energy * w.e) + ((1 - decay) * w.d));
}

/** Top-level: compute all three signals for a module at once. */
function computeSignals(mod, eventLog) {
  const primaryTypes = new Set(
    Object.entries(mod.events)
      .filter(([, v]) => v.primary)
      .map(([k]) => k)
  );
  const w      = mod.signal_functions.priority.weights || { e: 0.6, d: 0.4 };
  const energy = computeEnergy(eventLog, primaryTypes);
  const decay  = computeDecay(eventLog, primaryTypes);
  return { energy, decay, priority: computePriority(energy, decay, w) };
}
Signal Invariant
All three signal functions must be pure — they read only their eventLog argument and return a scalar. No external reads. No Date.now(). No random. Given the same log, they must always return the same values.
§ 06

Event Routing How excitement crosses VM boundaries without breaking determinism.

The event router is the connective tissue of the DDC system. It enforces the architectural rule: no shared memory, no direct function calls between VMs. The only inter-VM mechanism is an event appended to a target's log.

event-router.js — canonical cross-VM event envelope JS 
/**
 * Route an event from one VM to another.
 * The router is the ONLY path for inter-VM communication.
 *
 * Invariant: VM_stateₙ = deriveState(eventLog[0…n])
 * must hold for all VMs at all times.
 */
function routeEvent(targetVM, event) {
  const entry = Object.freeze({
    id:            makeEntryId(event),
    ts:            Date.now(),         // wall time — display only
    clock:         targetVM.clock++,   // logical clock — canonical ordering
    type:          event.type,
    origin:        event.origin,       // source VM's module_id
    payload:       event.payload,
    sig:           event.sig,          // ECDSA P-256 from originating identity
    correlationId: event.correlationId,// same chain across all VMs
    causationId:   event.id,           // the specific event that triggered this
    status:        "ok",
    cost:          computeCost(event)
  });

  targetVM.eventLog.push(entry);       // append only — never mutate existing
  targetVM.state = deriveState(targetVM.eventLog);
  targetVM.signals = computeSignals(targetVM.schema, targetVM.eventLog);
  emitEffects(targetVM, entry);        // render / network / metrics
}

/**
 * The router decides which VMs to notify based on:
 * 1. Explicit targetModule in payload
 * 2. Convergence: VMs with overlapping correlationId space + priority > θ
 */
function dispatch(sourceVM, event, allVMs, θ = 0.5) {
  // Explicit routing
  if (event.targetModule) {
    const target = allVMs.get(event.targetModule);
    if (target) routeEvent(target, event);
    return;
  }

  // Convergence routing — excitement attracts excited nodes
  for (const [id, vm] of allVMs) {
    if (id === sourceVM.schema.module_id) continue;
    if (vm.signals.priority < θ) continue;
    if (correlationOverlap(sourceVM, vm, event)) {
      routeEvent(vm, event);
    }
  }
}

App Builder / Spec Builder How the lava lamp concept becomes a runnable app scaffold.

The Planespace App Builder is the DDC system's spec-to-running-VM pipeline. Given a natural-language description and a module schema, it produces a fully hydrated, self-contained HTML app that embodies the DDC invariants: an immutable eventLog, a pure deriveState, and live signal functions.

The App Builder as DDC itself

💡

The App Builder is itself a DDC system. The user's prompt is the entropy source. It is injected as a single event. Ollama's generation is the deterministic reduction (given seed=42). The hydrated template is the rendered state. The download button is the emitted effect.

Sovereign Net OS Integration DDC as the operating model of a decentralised, browser-native OS.

Sovereign Net OS is the runtime that hosts DDC VMs in a fully decentralised, peer-to-peer environment. Every capability in the OS — identity, messaging, code editor, file storage, marketplace — maps to a DDC module with its own eventLog, deriveState, and signal functions.

ECDSA P-256 DIDs. Primary events: IdentityCreated, IdentityUpdated. Decay = staleness (security signal).

💬

Genesis/messaging

AES-256-GCM encrypted channels. Primary: MessageSent. Energy = message rate. Decay = thread silence.

📋

Genesis/tasks

Collaborative task management. Priority: energy-dominant (0.6/0.4). Convergence drives cross-user assignment.

🛒

Genesis/ecommerce

P2P marketplace. Primary: OrderPlaced. Energy-dominant (0.7/0.3). High-energy modules get priority routing.

Merkle-DAG versioning. Each commit is an event. rulesVersion pins the merge semantics for that branch.

Peer-to-Peer DDC

In Sovereign Net OS, each browser tab (or Electron node) is a host running its own set of DDC VMs. Cross-peer event routing uses BroadcastChannel (same machine) or WebRTC/WebSocket (across nodes), with the same envelope: correlationId, causationId, sig.

§ 11

System Invariants Rules that cannot be broken without leaving the DDC model.

I-1 · State Derivation
VM_stateₙ = deriveState(eventLog[0…n])
No external reads. No ambient globals. No time. No randomness. The log is the only input to state.
I-2 · Append-Only Log
Events are never mutated, reordered, or deleted after recording. History is immutable. Correction events are new appends, not retroactive edits.
I-3 · Pure Signal Functions
energy, decay, and priority are pure functions of the eventLog. Same log → same signals. They never call Date.now() or Math.random().
I-4 · No Hidden Channels
VMs communicate only through the event router. No shared memory. No direct function calls across VM boundaries. No global state.
I-5 · Entropy at the Boundary
Non-deterministic inputs (user events, network data, time, lava lamp entropy) are captured at the VM boundary as events. Once inside, they are fixed and replayable.
I-6 · rulesVersion Pins Semantics
deriveState with rulesVersion=N always produces the same output for the same log. Changing business rules requires incrementing rulesVersion, not silently modifying existing logic.
